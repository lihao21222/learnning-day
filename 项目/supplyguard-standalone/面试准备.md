# SupplyGuard 面试准备指南

---

## 一、项目介绍（3-5 分钟）

### 1.1 项目背景

> **面试话术**:
>
> SupplyGuard 是一个面向供应链风险监控任务的多 Agent 智能风控平台。
> 随着供应链越来越复杂，企业对供应商风险监控的需求也越来越迫切。
> 传统方法主要靠人工收集信息和评估，效率低且容易遗漏。
> 这个项目就是想用 AI 技术来提升供应链风控的效率和准确性。

---

## 二、系统架构

### 2.1 整体架构

```
┌───────────────────────────────────────────────┐
│              接入层 (Streamlit UI)             │
└───────────────────────────────────────────────┘
                        ↓
┌───────────────────────────────────────────────┐
│          Multi-Agent 编排器 (模拟 LangGraph)   │
│  ┌─────────────────────────────────────────┐ │
│  │ 数据收集 Agent → 风险分析 Agent         │ │
│  │     ↓              ↓                   │ │
│  │ 交叉验证 Agent → 报告生成 Agent         │ │
│  └─────────────────────────────────────────┘ │
└───────────────────────────────────────────────┘
                        ↓
┌───────────────────────────────────────────────┐
│               知识库与数据层                    │
│  ┌─────────────────────────────────────────┐ │
│  │ 供应商数据 | 风险规则 | 行业动态       │ │
│  └─────────────────────────────────────────┘ │
└───────────────────────────────────────────────┘
```

## 三、核心模块详解

### 3.1 Multi-Agent 系统

| Agent | 职责 | 设计思路 |
|------|------|---------|
| **数据收集 Agent** | 从多源收集供应商信息 | 聚合分散的数据源 |
| **风险分析 Agent** | 识别与评估风险 | 基于规则与语义匹配 |
| **交叉验证 Agent** | 验证风险信息准确性 | 提升可信度 |
| **报告生成 Agent** | 生成结构化报告 | 便于用户理解 |

> **面试话术**:
>
> 我用策略模式设计了这四个 Agent，每个 Agent 职责单一，便于扩展和维护。
> 编排器按照固定的工作流依次执行每个 Agent，有点类似于 LangGraph 的设计思路。
> 这样做的好处是每个步骤都清晰可控，而且审计起来很方便。

---

### 3.2 风险评估机制

**风险等级**:
- 低风险 (0-20分)
- 中风险 (20-40分)
- 高风险 (40-60分)
- 严重风险 (60分以上)

**评分逻辑**:
- 每个风险类型有不同权重
- 关键词匹配数量影响置信度
- 综合计算整体风险分数

---

### 3.3 Human-in-the-Loop (人工干预)

**设计思路**:
- AI 先做初步评估
- 关键风险点需要人工确认
- 人工反馈可以用来优化模型

> **面试话术**:
>
> 风控问题比较敏感，完全让 AI 做决策是不够的。
> 所以我设计了人工干预机制，系统会先给出建议，
> 最终决策可以由人来做，而且人的判断也能用来优化系统。

## 四、常见面试问题与答案

### Q1: 为什么用 Multi-Agent 架构，而不是单一 Agent？

**参考答案**:
> 用 Multi-Agent 有几个好处：
>
> 1. **职责分离** - 每个 Agent 专门做一件事，代码更清晰
> 2. **可扩展性** - 想加新功能可以加新 Agent，不影响现有
> 3. **可观测性** - 每个 Agent 的输出都能看到，便于调试和审计
> 4. **并行化** - 将来可以让某些 Agent 并行执行，提升效率
>
> 这个设计类似于 LangGraph，不过我简化了一些，核心思想是一样的。

---

### Q2: 怎么避免误判和漏判？

**参考答案**:
> 我从几个方面来处理：
>
> 1. **交叉验证** - 让不同 Agent 或不同数据源交叉验证
> 2. **多重证据** - 要有多个证据才会触发高风险
> 3. **人工确认** - 关键风险必须人工确认
> 4. **风险分数** - 不是简单的是或否，而是给一个分数，让用户判断
>
> 另外，系统会记录所有操作日志，方便回溯分析误判漏判的原因。

---

### Q3: 怎么评估这个系统的效果？

**参考答案**:
> 可以从以下几个维度评估：
>
> 1. **准确率** - 识别出来的风险真的是风险的比例
> 2. **召回率** - 真正的风险被识别出来的比例
> 3. **误报率** - 误报的频率
> 4. **人工审核效率** - 和纯人工比，节省多少时间
>
> 实际落地时可以做 A/B 测试，对比用和不用系统的效果差异。

---

### Q4: 下一步想优化什么？

**参考答案**:
> 我有几个优化方向：
>
> 1. **集成真实 LLM** - 现在是规则匹配，可以加本地 Ollama 或 API
> 2. **向量数据库** - 用 Qdrant 或 Chroma 做语义检索
> 3. **知识图谱** - 用 Neo4j 存储供应链关系，可以做关联分析
> 4. **服务化** - 用 FastAPI 做服务端，支持更多集成场景
> 5. **自适应学习** - 根据人工反馈自动调整规则权重

---

### Q5: 遇到的最大挑战是什么？

**参考答案**:
> 我觉得最大的挑战是**平衡自动化和可信度**。
>
> 完全自动化用户不敢用，完全人工效率低。
> 我在 Human-in-the-Loop 机制上花了很多心思，
> 让 AI 做辅助决策，人做最终判断，同时人的判断能帮助系统优化。
>
> 另外，Agent 之间的数据流转和状态管理也需要仔细设计。

---

## 五、项目亮点总结

1. ✅ **Multi-Agent 架构** - 展示复杂系统设计能力
2. ✅ **可审计的决策** - 完整审计日志，每个步骤可追溯
3. ✅ **Human-in-the-Loop** - 人机结合的实际考虑
4. ✅ **可视化监控** - Streamlit 仪表板直观展示
5. ✅ **风险量化** - 科学的评分机制
6. ✅ **知识库设计** - 知识组织与管理
7. ✅ **工程化实现** - 配置、日志、分层设计

## 六、推荐展示流程

面试时推荐按以下顺序展示：

1. **Web 界面演示** - 先打开 Streamlit 仪表板，视觉效果好
2. **快速走查** - 用 demo.py 演示风险评估流程
3. **代码讲解** - 重点讲 agents.py 和整体架构
4. **设计思路** - 讲为什么这么设计，有什么权衡
5. **未来规划** - 展示自己的思考深度
