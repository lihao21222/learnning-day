## 1. 项目介绍类

---

### Q1: 请简单介绍一下你的这个项目？

**参考答案：**

这是一个本地知识库问答系统，也就是 RAG（检索增强生成）项目。它允许用户上传自己的文档（支持 TXT、PDF、Word 等多种格式），然后可以像和 AI 对话一样询问文档中的内容。

项目有几个特点：
1. **多格式支持**：支持 TXT、PDF、DOCX 文档
2. **分层架构**：从简化版、进阶版到专业版，适合不同场景
3. **模块化设计**：文档加载、文本分割、检索等模块解耦
4. **Web 界面**：提供了 Streamlit 可视化界面，演示效果好

这个项目可以用于企业内部知识库、个人学习助手、文档问答等场景。

---

### Q2: 为什么想做这个项目？

**参考答案：**

有几个原因：

1. **技术学习**：RAG 是目前 NLP 领域的热点，想通过项目深入理解
2. **实用性**：自己平时也有大量文档需要整理和查询，这个工具很实用
3. **面试准备**：完整的 RAG 项目可以展示全栈开发能力
4. **可扩展性**：这个项目有很多优化空间，可以持续迭代

---

### Q3: 这个项目的难点是什么？

**参考答案：**

主要难点有：

1. **文档格式多样性**：不同格式的文档解析方式不同，需要设计好架构
2. **检索准确性**：简单的关键词匹配不够，需要考虑语义检索
3. **用户体验**：如何让问答过程更流畅自然
4. **可扩展性**：如何设计好架构，方便后续添加新功能

## 2. 技术架构类

---

### Q4: 请介绍一下你的项目架构？

**参考答案：**

项目采用分层架构，主要分为以下几个模块：

```
┌─────────────────────────────────────────┐
│        交互层 (UI)                       │
│  (命令行 / Streamlit Web 界面)           │
└────────────────────┬────────────────────┘
                     │
┌────────────────────▼────────────────────┐
│        RAG 问答系统                      │
│  (查询处理 / 结果整理)                   │
└────────────────────┬────────────────────┘
                     │
    ┌────────────────┴────────────────┐
    │                                 │
┌───▼───────────┐           ┌─────────▼────────┐
│   检索器      │           │   文档加载器      │
└───┬───────────┘           └─────────┬────────┘
    │                                 │
┌───▼───────────┐           ┌─────────▼────────┐
│  文本分割器   │           │  多格式解析器     │
└───────────────┘           └──────────────────┘
```

**核心模块：**
1. **文档加载器**：`document_loader_pro.py`，负责加载和解析不同格式的文档
2. **文本分割器**：将长文档分割成合适大小的块
3. **检索器**：根据查询找到相关的文档片段
4. **问答系统**：整合检索结果，生成友好的回答

---

### Q5: 为什么这样设计架构？

**参考答案：**

1. **模块化**：每个模块功能单一，便于维护和测试
2. **可扩展**：比如可以轻松添加新的文档格式解析器
3. **可替换**：各个模块可以独立升级，比如把关键词检索换成向量检索
4. **分层清晰**：从底层文档处理到上层交互，职责明确

---

### Q6: 你的项目有几个版本？为什么这样设计？

**参考答案：**

项目有三个主要版本：

1. **简化版**（`simple_rag.py`）：纯 Python，无外部依赖，适合快速上手和学习核心逻辑
2. **进阶版**（`advanced_rag.py`）：工程化，含配置和日志系统，适合展示代码规范
3. **专业版**（`rag_pro.py`）：多格式支持，策略模式设计，适合面试演示

这样设计的好处是：
- **逐步递进**：可以从简单到复杂逐步讲解
- **按需选择**：不同场景用不同版本
- **对比展示**：可以展示从原型到工程化的演进过程

## 3. RAG 技术类

---

### Q7: 什么是 RAG？为什么需要 RAG？

**参考答案：**

RAG 是 Retrieval-Augmented Generation（检索增强生成）的缩写。

**为什么需要 RAG？**

1. **解决幻觉问题**：大模型有时会编造事实，RAG 让模型基于真实文档回答
2. **知识更新**：不需要重新训练模型，只需更新知识库
3. **隐私保护**：数据在本地，不需要上传到外部 API
4. **成本降低**：相比微调大模型，RAG 的成本更低

---

### Q8: RAG 的工作原理是什么？

**参考答案：**

RAG 的工作流程分为两个阶段：

**索引阶段（Indexing）：**
1. 加载文档
2. 将文档分割成小的片段（Chunks）
3. 将每个片段向量化（Embedding）
4. 存储到向量数据库

**检索阶段（Retrieval）：**
1. 将用户问题向量化
2. 在向量数据库中找到最相似的 Top-K 个文档片段
3. 将检索到的文档片段和问题一起传给 LLM
4. LLM 基于检索结果生成回答

---

### Q9: 你的 RAG 系统和真实的 RAG 有什么区别？

**参考答案：**

我的项目是一个简化版的 RAG，主要区别是：

1. **检索方式**：真实 RAG 用向量检索（语义相似度），我目前用的是关键词匹配
2. **LLM 集成**：真实 RAG 会调用大模型生成回答，我目前是用模板整理检索结果
3. **向量数据库**：真实项目常用 Chroma、Pinecone 等向量数据库

但核心的 RAG 思想都有了：**先检索，后回答**，后续可以逐步升级。

---

### Q10: 为什么需要把文档分割成块？

**参考答案：**

1. **上下文长度限制**：大多数 LLM 有输入长度限制（比如 4K、8K、16K tokens）
2. **检索精度**：太长的文档会导致检索精度下降，小块更容易匹配到相关内容
3. **计算效率**：小片段的向量化和检索更快

---

### Q11: 如何选择块大小（chunk_size）？

**参考答案：**

块大小的选择需要平衡几个因素：

1. **文档类型**：代码文档可能需要更小的块，而自然语言可以稍大
2. **检索精度**：小块精度高，但可能丢失上下文
3. **LLM 能力**：上下文窗口大的模型可以用更大的块

**常见的块大小：**
- 256-512 tokens：适合精确检索
- 512-1024 tokens：平衡选择
- 1024+ tokens：适合长文档理解

我的项目用了 500 字符作为默认值，是一个比较中庸的选择。

## 4. 代码实现类

---

### Q12: 请介绍一下 document_loader_pro.py 的设计？

**参考答案：**

这是一个用策略模式设计的文档加载器：

1. **基类**：`BaseDocumentParser` 定义接口
2. **具体策略**：
   - `TextParser`：处理 TXT/MD 文件
   - `PDFParser`：处理 PDF 文件
   - `DocxParser`：处理 DOCX 文件
3. **上下文类**：`DocumentLoaderPro` 统一调度

这样设计的好处是：
- 开闭原则：添加新格式只需新增一个 Parser 类
- 代码复用：各个 Parser 独立开发和测试
- 优雅降级：某个 Parser 的依赖缺失不影响其他功能

---

### Q13: 文本分割器是如何实现的？

**参考答案：**

有两个版本的实现：

**进阶版（`advanced_rag.py`）**：
- 最初尝试按句子分割，用句号、感叹号等作为分隔符
- 但后来发现会修改原文，所以改成了按字符数固定分割

**专业版（`rag_pro.py`）**：
- 按字符数简单分割（chunk_size=500）
- 保留重叠部分（chunk_overlap=50）
- 这样既简单又能保持一定的上下文连续性

**为什么最后选择按字符数分割？**
- 简单可靠，不会因为特殊字符出错
- 保持原文内容不变，不破坏文档结构

---

### Q14: 检索器是如何实现的？

**参考答案：**

目前的检索器是关键词匹配：

1. 将查询转成小写
2. 遍历所有文档块，计算得分：
   - 完整匹配：+100 分
   - 分词匹配：每个词 +10 分
3. 按得分排序，取 Top-K

**优点：** 简单、快速、不需要外部依赖

**局限性：** 只能匹配字面，不能理解语义

**后续优化方向：** 集成 Sentence-Transformers 做向量检索

---

### Q15: 配置文件（config.py）是怎么设计的？

**参考答案：**

配置文件使用简单的常量定义：

```python
KNOWLEDGE_BASE_DIR = "knowledge_base"
CHUNK_SIZE = 500
CHUNK_OVERLAP = 50
TOP_K = 3
```

这样设计的原因：
1. **集中管理**：所有配置在一个地方，便于修改
2. **简单直观**：不需要复杂的配置解析库
3. **类型提示**：配合 Python 的类型注解，更安全

---

### Q16: 日志系统（logger.py）是怎么实现的？

**参考答案：**

使用 Python 标准库的 logging 模块：

1. **同时输出到文件和控制台**
2. **按日期分割日志文件**
3. **统一的日志格式**：`时间 - 名称 - 级别 - 消息`

日志的作用：
- 调试问题时可以查看历史记录
- 了解系统运行状态
- 性能分析（比如文档加载耗时）

## 5. 设计模式类

---

### Q17: 项目中用到了哪些设计模式？

**参考答案：**

1. **策略模式**：`document_loader_pro.py` 中的文档解析器
2. **模板方法**（简化版）：文本分割的流程是固定的
3. **单例模式**（隐式）：`config.py` 的全局配置
4. **门面模式**：`RAGSystemPro` 封装了复杂的子系统

---

### Q18: 请详细讲一下策略模式的实现？

**参考答案：**

在 `document_loader_pro.py` 中：

1. **抽象策略**：`BaseDocumentParser`
   ```python
   class BaseDocumentParser(ABC):
       @abstractmethod
       def parse(self, file_path):
           pass
   ```

2. **具体策略**：`TextParser`, `PDFParser`, `DocxParser`

3. **上下文**：`DocumentLoaderPro`
   - 维护所有策略的实例
   - 根据文件扩展名选择合适的策略

**为什么用策略模式？**
- 需要在运行时根据文件类型选择不同的解析方式
- 避免大量的 `if-elif-else`
- 方便添加新的文档格式

---

### Q19: 策略模式有什么优缺点？

**参考答案：**

**优点：**
1. 开闭原则：添加新策略不需要修改上下文
2. 避免多重条件语句
3. 算法可以独立演化

**缺点：**
1. 增加类的数量
2. 调用方需要了解策略的区别

## 6. 优化扩展类

### Q20: 如果让你优化这个项目，你会怎么做？

**参考答案：**

我会从以下几个方面优化：

1. **检索升级**：
   - 集成 Sentence-Transformers 做向量检索
   - 使用 Chroma 或 FAISS 作为向量数据库

2. **LLM 集成**：
   - 通过 Ollama 集成本地 LLM（比如 Llama 3、Qwen）
   - 或者调用 OpenAI API（配置好代理）

3. **功能增强**：
   - 添加对话记忆，支持多轮对话
   - 支持文档更新和删除
   - 添加引用来源的高亮显示

4. **性能优化**：
   - 并行文档处理
   - 向量检索的缓存
   - 异步查询处理

5. **界面优化**：
   - Streamlit 界面美化
   - 添加对话历史展示
   - 支持文档预览

---

### Q21: 如何集成向量检索？

**参考答案：**

大概步骤：

1. **安装依赖**：
   ```bash
   pip install sentence-transformers chromadb
   ```

2. **索引阶段**：
   - 用 Sentence-Transformers 生成向量
   - 存储到 Chroma

3. **检索阶段**：
   - 将查询也转成向量
   - 在 Chroma 中做相似度搜索

4. **代码修改**：
   - 新建一个 `VectorRetriever` 类
   - 保持接口不变，方便替换

---

### Q22: 如何集成 LLM？

**参考答案：**

有几种方式：

**方案一：本地 LLM（推荐用于面试演示）**
- 使用 Ollama
- 好处：数据本地，演示真实
- 示例：
  ```python
  from langchain.llms import Ollama
  llm = Ollama(model="llama3")
  ```

**方案二：API 调用（更灵活）**
- OpenAI / Claude / 智谱等
- 需要 API Key

**RAG 提示词示例：**
```
你是一个问答助手。请根据以下上下文回答问题。

上下文：
{检索到的文档内容}

问题：
{用户问题}

请仅根据上下文回答，不要编造内容。
```

---

### Q23: 如何支持多轮对话？

**参考答案：**

需要添加对话记忆功能：

1. **存储对话历史**：
   - 使用列表保存对话上下文
   - 限制历史长度，避免上下文过长

2. **上下文压缩**：
   - 只保留最近的 K 轮对话
   - 或者用摘要代替

3. **查询改写**：
   - 根据历史对话改写当前查询
   - 让查询更完整

## 7. 困难挑战类

---

### Q24: 做这个项目时遇到了什么困难？怎么解决的？

**参考答案：**

**困难 1：文本分割器的设计**

- 最开始尝试按句子分割，但发现处理不好中文标点和各种边界情况
- 而且还会修改原文，添加额外的句号
- **解决方案**：改用按字符数固定分割，简单可靠

**困难 2：多格式文档解析**

- PDF 和 Word 需要安装额外依赖
- 不同用户的环境不一样
- **解决方案**：设计成策略模式，支持优雅降级，依赖缺失提示安装

**困难 3：Windows 控制台编码问题**

- Emoji 和特殊字符在 Windows 控制台会报错
- **解决方案**：创建不使用 emoji 的演示版本，用文字标签代替

---

### Q25: 如果检索结果不相关怎么办？

**参考答案：**

可以从几个方面解决：

1. **优化检索**：
   - 改用向量检索
   - 调整块大小和重叠
   - 使用混合检索（关键词 + 向量）

2. **重排序**：
   - 先用宽松的条件召回更多候选
   - 再用小模型做重排序

3. **查询改写**：
   - 用 LLM 改写或扩展用户查询

4. **用户反馈**：
   - 让用户可以点击"这不是我要的"
   - 收集反馈持续优化

---

### Q26: 如何评估 RAG 系统的好坏？

**参考答案：**

可以从几个维度评估：

1. **检索质量**：
   - 准确率（检索到的文档是否相关）
   - 召回率（相关的文档是否都检索到了）

2. **回答质量**：
   - 正确性（回答是否准确）
   - 完整性（是否覆盖所有要点）
   - 有用性（对用户是否有帮助）

3. **用户体验**：
   - 响应速度
   - 界面友好度
   - 交互流畅度

---

### Q27: 你的项目和 LangChain 有什么关系？

**参考答案：**

**当前关系：**
- 目前没有使用 LangChain

---

### Q28: 如果文档非常大（比如 1GB），会有什么问题？怎么解决？

**参考答案：**

**可能的问题：**
1. 内存占用大
2. 加载和索引慢
3. 检索效率低

**解决方案：**

1. **增量索引**：
   - 只在文档变更时重新处理
   - 记录每个文件的处理状态

2. **并行处理**：
   - 多线程/多进程并行处理文档
   - 使用任务队列（Celery / RQ）

3. **向量数据库优化**：
   - 使用 HNSW 等索引加速
   - 分布式向量数据库

4. **分页加载**：
   - Web 端可以分页展示
   - 不用一次性加载所有

5. **承认不足**：但要紧接着说优化思路

### 常见问题的回答策略：

| 问题类型 | 回答策略 |
|----------|----------|
| "介绍一下你的项目" | 1-2 分钟讲清楚背景、功能、亮点 |
| "遇到什么困难" | 说 1-2 个真实困难，重点讲解决思路 |
| "怎么优化" | 分点说，按优先级排序 |
| "和 XX 有什么区别" | 客观承认差异，但说明自己项目的价值 |
