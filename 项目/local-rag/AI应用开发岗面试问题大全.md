## 1. RAG 技术类

---

### Q1: 请详细介绍 RAG 的完整工作流程

**参考答案：**

RAG（检索增强生成）的完整流程包括两个阶段：

**索引阶段（Indexing）：**
1. 文档加载（Load）：从各种来源（PDF、Word、网页等）加载文档
2. 文档分割（Split）：将长文档分割成合适大小的块（Chunks）
3. 向量化（Embed）：用 Embedding 模型将每个文本块转成向量
4. 存储（Store）：将向量和原始文本存入向量数据库

**查询阶段（Querying）：**
1. 查询向量化：将用户问题转成向量
2. 相似性检索：在向量数据库中找到 Top-K 个最相关的文本块
3. 上下文构建：将检索到的文本块和问题组合成提示词
4. LLM 生成：将提示词发给 LLM，生成最终回答

---

### Q2: RAG 和 Fine-tuning（微调）有什么区别？什么时候用哪个？

**参考答案：**

| 对比维度 | RAG | 微调 |
|---------|-----|------|
| **数据更新** | 只需更新知识库，快速 | 重新训练，慢 |
| **成本** | 低（只需 Embedding 和检索） | 高（需要 GPU 和数据） |
| **幻觉问题** | 较小（基于事实文档） | 可能存在 |
| **知识新鲜度** | 高（可随时更新） | 固定在训练数据截止日期 |
| **可解释性** | 好（可以看到来源） | 较差（黑盒） |
| **领域适配** | 适合事实类问答 | 适合风格、格式类任务 |

**如何选择？**
- 需要频繁更新知识 → 用 RAG
- 事实性要求高 → 用 RAG
- 需要风格迁移、格式转换 → 用微调
- 两者结合 → RAG + 微调（RAG 做事实检索，微调做风格对齐）

---

### Q3: 常见的 RAG 优化策略有哪些？

**参考答案：**

**1. 检索优化：**
- 混合检索（Hybrid Search）：结合关键词和向量
- 重排序（Rerank）：先用宽召回，再用小模型重排序
- 查询改写（Query Rewriting）：用 LLM 改写或扩展查询
- 分块优化：调整 chunk_size 和 overlap

**2. 生成优化：**
- 提示词工程（Prompt Engineering）
- 思维链（Chain-of-Thought）
- 检索增强的提示词

**3. 架构优化：**
- 多级检索（Coarse-to-Fine）
- 查询压缩
- 缓存机制

**4. 评估优化：**
- 用 LLM 做评估
- A/B 测试
- 人类评估

---

### Q4: 什么是 RAG 的"幻觉"？如何缓解？

**参考答案：**

**RAG 幻觉的两种类型：**
1. **检索错误**：检索到了不相关的文档
2. **生成错误**：LLM 基于错误的上下文编造内容

**缓解方法：**

1. **检索端：**
   - 提高检索准确率
   - 使用混合检索 + 重排序
   - 增加返回的文档数量

2. **生成端：**
   - 提示词强调"仅根据上下文回答"
   - 让 LLM 在不确定时说"不知道"
   - 在回答中标注引用来源

3. **系统层面：**
   - 人工反馈闭环
   - 持续评估和优化

---

### Q5: RAG 评估指标有哪些？

**参考答案：**

**检索指标：**
- 准确率（Precision）：检索结果中相关的比例
- 召回率（Recall）：相关结果被检索到的比例
- F1 值：准确率和召回率的调和平均
- MRR（Mean Reciprocal Rank）：第一个相关结果的倒数排名

**端到端指标：**
- 正确性（Answer Correctness）：回答是否准确
- 有用性（Helpfulness）：回答对用户是否有帮助
- 忠实度（Faithfulness）：回答是否基于上下文，没有编造
- 上下文关联度（Context Relevance）：检索到的上下文是否相关

**评估方法：**
- LLM 自动评估（GPT-4 做裁判）
- 人类评估
- 混合评估（LLM + 人工）

## 2. LangChain 技术类

---

### Q6: 介绍一下 LangChain 及其核心组件？

**参考答案：**

LangChain 是一个用于构建 LLM 应用的框架。

**核心组件：**

1. **Models（模型）**：
   - LLM：大语言模型
   - Chat Models：对话模型
   - Embedding Models：嵌入模型

2. **Prompts（提示词）**：
   - Prompt Templates：提示词模板
   - FewShotPromptTemplate：少样本提示词
   - ChatPromptTemplate：对话提示词

3. **Indexes（索引）**：
   - Document Loaders：文档加载器
   - Text Splitters：文本分割器
   - Vector Stores：向量存储
   - Retrievers：检索器

4. **Chains（链）**：
   - LLMChain：基础链
   - Sequential Chains：顺序链
   - Router Chains：路由链

5. **Agents（代理）**：
   - 让 LLM 自己决定下一步做什么
   - 可以使用工具

6. **Memory（记忆）**：
   - Conversation Buffer Memory
   - Summary Memory

---

### Q7: LangChain 中的 Chain 和 Agent 有什么区别？

**参考答案：**

| 维度 | Chain | Agent |
|------|-------|-------|
| **执行方式** | 预定义好的固定流程 | 让 LLM 动态决定下一步 |
| **灵活性** | 较低，按预设执行 | 高，根据情况变化 |
| **适用场景** | 固定流程的任务 | 复杂、不确定的任务 |
| **可控性** | 高，知道每一步做什么 | 较低，LLM 自主决策 |
| **例子** | 文档问答链 | 数学计算、网页搜索 |

**简单理解：**
- Chain 像流水线，步骤固定
- Agent 像人，会思考和决策

---

### Q8: 你用过 LangChain 的哪些 Document Loaders？

**参考答案：**

**常用的 Loaders：**
1. 文本类：`TextLoader`, `DirectoryLoader`
2. PDF：`PyPDFLoader`, `PyPDFium2Loader`, `PDFMinerLoader`
3. Word：`Docx2txtLoader`, `UnstructuredWordDocumentLoader`
4. 网页：`WebBaseLoader`, `SeleniumURLLoader`
5. 社交媒体：`NotionLoader`, `TwitterLoader`
6. 数据库：`SQLDatabase`

**选择建议：**
- 简单 PDF → PyPDFLoader
- 复杂 PDF → PyPDFium2Loader
- 通用格式 → UnstructuredXXLoader

---

### Q9: LangChain 有哪些 Text Splitter？如何选择？

**参考答案：**

**常用的 Text Splitter：**

| Splitter | 特点 | 适用场景 |
|----------|------|----------|
| `CharacterTextSplitter` | 按字符数分割 | 简单通用 |
| `RecursiveCharacterTextSplitter` | 递归按分隔符分割（推荐） | 自然语言 |
| `TokenTextSplitter` | 按 token 数分割 | 精确控制 |
| `MarkdownTextSplitter` | 感知 Markdown 结构 | Markdown 文档 |
| `CodeTextSplitter` | 感知代码结构 | 代码文档 |
| `SentenceTransformersTokenTextSplitter` | 配合 Embedding 模型 | 语义检索 |

**选择建议：**
- 一般推荐 `RecursiveCharacterTextSplitter`
- 分隔符：`["\n\n", "\n", "。", ".", " ", ""]`
- chunk_size：256-1024 tokens
- chunk_overlap：20-30%

---

### Q10: 如何在 LangChain 中实现对话记忆？

**参考答案：**

**常用的 Memory 类型：**

1. **ConversationBufferMemory**（最简单）：
   - 保存完整对话历史
   - 适合短对话

2. **ConversationBufferWindowMemory**：
   - 只保存最近的 K 轮对话
   - 适合长对话

3. **ConversationSummaryMemory**：
   - 定期将历史对话总结
   - 节省 token，适合非常长的对话

4. **ConversationSummaryBufferMemory**：
   - 混合：最近几轮保存原文，更早的保存总结

**使用示例：**
```python
from langchain.memory import ConversationBufferMemory
from langchain.chains import ConversationChain

memory = ConversationBufferMemory()
chain = ConversationChain(llm=llm, memory=memory)
```

## 3. 大模型应用类

---

### Q11: 你对 Prompt Engineering 有什么理解？

**参考答案：**

提示词工程是通过设计和优化提示词来引导 LLM 更好地完成任务的过程。

**核心原则：**

1. **清晰明确**：指令要具体，不要模棱两可
2. **结构化**：使用分隔符（---, ```, 等）区分不同部分
3. **给示例**：Few-Shot Learning，给几个示例
4. **要求推理**：让模型"一步步思考"

**常用技巧：**

- **角色扮演**："你是一个专家..."
- **思维链**："请一步步思考..."
- **格式指定**："请用 JSON 格式返回..."
- **约束条件**："仅根据上下文回答..."

**提示词框架：**

```
角色：你是 [角色描述]
任务：你需要 [任务描述]
上下文：[相关信息]
要求：[具体要求]
输出格式：[格式要求]
```

---

### Q12: 什么是 CoT（思维链）？什么时候用？

**参考答案：**

**CoT（Chain-of-Thought，思维链）**：让 LLM 在回答问题时，先展示思考过程，再给出最终答案。

**简单的 CoT 提示词：**
> "请一步步思考，然后给出答案。"

**Few-Shot CoT**：给几个包含推理过程的示例。

**什么时候用 CoT？**

- ✅ 数学题、逻辑题
- ✅ 需要多步推理的任务
- ✅ 复杂决策问题
- ❌ 简单问答（用了反而变慢）

**进阶：Zero-Shot CoT**
> "让我们一步步思考。"
> 不需要给示例，也能提升推理能力

---

### Q13: 如何降低 LLM 应用的成本？

**参考答案：**

**成本构成：**
- 输入 token + 输出 token

**优化策略：**

1. **提示词优化**：
   - 精简提示词，删除不必要的内容
   - 提示词压缩
   - 用更短的指令

2. **模型选择**：
   - 小任务用小模型（GPT-3.5 → Qwen/Qianfan）
   - 分级路由：简单问题用便宜模型，复杂问题用贵模型

3. **缓存**：
   - 相同问题直接返回缓存结果
   - Semantic Cache（语义缓存）

4. **输出控制**：
   - max_tokens 限制
   - 要求简洁回答

5. **批处理**：
   - 多个问题合并处理

---

### Q14: 大模型应用如何处理长上下文？

**参考答案：**

**方案一：RAG（推荐）**
- 优点：不需要太长的上下文窗口
- 适用：需要从大量文档中找信息

**方案二：长上下文模型**
- GPT-4 Turbo（128K）、Claude 3（200K）
- 优点：简单
- 缺点：贵，速度慢

**方案三：层次化处理**
1. 先将长文档分成多个部分
2. 让 LLM 分别阅读每一部分，提取关键点
3. 整合各部分的关键点，形成最终回答

**方案四：提示词压缩**
- 去掉冗余内容
- 用摘要代替原文

---

### Q15: 如何做 LLM 的输出解析？

**参考答案：**

**方法一：提示词指定格式**
> "请用 JSON 格式返回，包含以下字段：..."

**方法二：使用 LangChain Output Parsers**

常用的 Parser：
- `PydanticOutputParser`：用 Pydantic 模型定义
- `JsonOutputParser`：JSON 解析
- `CommaSeparatedListOutputParser`：逗号分隔列表
- `DatetimeOutputParser`：日期时间

**示例：**
```python
from langchain.output_parsers import PydanticOutputParser
from pydantic import BaseModel, Field

class Answer(BaseModel):
    answer: str = Field(description="答案")
    confidence: float = Field(description="置信度")

parser = PydanticOutputParser(pydantic_object=Answer)
```

**方法三：容错处理**
- 先尝试严格解析
- 失败时用 LLM 修复
- 或降级到宽松解析

## 4. 向量数据库类

---

### Q16: 什么是向量数据库？和传统数据库有什么区别？

**参考答案：**

**向量数据库**：专门用于存储和检索向量的数据库。

**主要区别：**

| 维度 | 传统数据库 | 向量数据库 |
|------|-----------|-----------|
| **数据类型** | 结构化数据（表、列） | 向量（Embedding） |
| **查询方式** | SQL、精确匹配、范围查询 | 相似性搜索（Top-K） |
| **索引结构** | B 树、Hash 等 | HNSW、IVF、Annoy 等 |
| **应用场景** | 事务处理、业务数据 | 语义搜索、推荐、RAG |

**核心概念：**
- **Embedding**：将文本转成高维向量
- **相似度**：通常用余弦相似度或欧氏距离
- **ANN**：Approximate Nearest Neighbor，近似最近邻（为了速度牺牲一点精度）

---

### Q17: 常用的向量数据库有哪些？如何选择？

**参考答案：**

| 向量数据库 | 特点 | 适用场景 |
|-----------|------|----------|
| **Chroma** | 轻量级、本地、LangChain 集成好 | 开发、原型 |
| **FAISS** | 快、Facebook 开发、内存型 | 性能敏感、不持久化 |
| **Pinecone** | 托管服务、SaaS | 生产环境、不想运维 |
| **Weaviate** | 混合搜索（向量 + 关键词）、开源 | 生产环境、自托管 |
| **Milvus / Zilliz** | 企业级、功能全 | 大规模、生产环境 |
| **Qdrant** | 快、开源、支持过滤 | 生产环境、自托管 |

**选择建议：**
- 开发测试 → Chroma
- 简单快速 → FAISS
- 生产环境不想运维 → Pinecone
- 生产环境自托管 → Weaviate / Qdrant / Milvus

---

### Q18: 什么是 HNSW？什么是 IVF？

**参考答案：**

**ANN 算法（近似最近邻）**：为了速度，不找精确的最近邻，找近似的。

**HNSW（Hierarchical Navigable Small World）**：
- 类似多层跳表
- 顶层稀疏，底层稠密
- 搜索时先在顶层快速定位到大概区域，再逐层细化
- 优点：查询快，支持增量添加
- 缺点：内存占用大

**IVF（Inverted File）**：
- 先用聚类将向量分成多个组
- 查询时先找最近的几个组，再在组内精确搜索
- 优点：内存占用小，适合大数据量
- 缺点：需要构建索引，不适合频繁更新

**如何选择？**
- 查询性能优先 → HNSW
- 大数据量、内存有限 → IVF
- 混合 → HNSW + IVF

---

### Q19: 如何选择 Embedding 模型？

**参考答案：**

**常用模型：**

| 模型 | 维度 | 特点 |
|------|------|------|
| `text-embedding-3-small` | 1536 | OpenAI、均衡 |
| `text-embedding-3-large` | 3072 | OpenAI、高质量 |
| `text-embedding-ada-002` | 1536 | OpenAI、经典 |
| `all-MiniLM-L6-v2` | 384 | Sentence-Transformers、轻量 |
| `all-mpnet-base-v2` | 768 | Sentence-Transformers、平衡 |
| `bge-large-zh-v1.5` | 1024 | 中文、BAAI、推荐 |

**选择建议：**

1. **语言**：中文优先考虑 bge、Qwen 等中文优化模型
2. **质量 vs 速度**：质量选大模型，速度选小模型
3. **成本**：
   - 开发测试 → 本地开源模型
   - 生产 → 根据预算选择
4. **一致性**：Embedding 模型不能随便换，要保持一致

---

### Q20: 向量检索准确率不够怎么办？

**参考答案：**

**1. 分块优化：**
- 调整 chunk_size（太小缺上下文，太大噪声多）
- 调整 chunk_overlap（20-30%）
- 尝试按语义分割（句子、段落）

**2. 检索优化：**
- 混合检索（向量 + BM25/关键词）
- 增加 K 值（召回更多候选）
- 查询改写（用 LLM 改写或扩展查询）

**3. 重排序（Rerank）**：
- 用 Cross-Encoder 做重排序
- 或用 LLM 做重排序
- BM25 召回前 20，重排序后取前 5

**4. Embedding 优化：**
- 尝试更好的 Embedding 模型
- 对特定领域微调 Embedding

**5. 查询压缩**：
- 让 LLM 从对话历史中提取真实问题

5. **可扩展性**：异步、队列

### Q22: 如何处理 LLM API 的限流和错误？

**参考答案：**

**限流处理：**

1. **限流识别**：
   - HTTP 429 状态码
   - Rate limit 错误信息

2. **重试策略**：
   - 指数退避（Exponential Backoff）
   - 抖动（Jitter）：避免重试风暴
   - 最大重试次数

3. **速率限制**：
   - 客户端限流器（如 Tenacity）
   - 令牌桶算法

4. **批处理**：
   - 将多个请求合并成一个批处理请求

**错误处理：**

1. **常见错误类型**：
   - 超时 → 重试
   - 限流 → 等待后重试
   - 内容违规 → 返回友好提示
   - 服务不可用 → 降级

2. **降级策略**：
   - 用小模型/本地模型替代
   - 返回缓存结果
   - 返回友好提示

---

### Q23: LLM 应用如何做缓存？

**参考答案：**

**缓存策略：**

1. **精确匹配缓存**：
   - 相同问题 → 返回缓存结果
   - 适合 FAQ、常见问题

2. **语义缓存（Semantic Cache）**：
   - 相似问题 → 相同答案
   - 用 Embedding 计算相似度
   - 相似度 > 阈值 → 命中缓存

3. **Cache Key 设计**：

## 7. 算法基础类

### Q29: 什么是 Embedding？如何计算两个文本的相似度？

**参考答案：**

**Embedding**：将文本（或其他数据）映射成高维空间中的向量（稠密向量），意思相近的文本在向量空间中距离近。

**常见的相似度计算方式：**

1. **余弦相似度（Cosine Similarity）**：
   - 计算两个向量的夹角余弦
   - 范围：[-1, 1]，1 最相似，0 不相关，-1 相反
   - 对向量长度不敏感（已经归一化时等价于点积）
   - **公式**：
     ```
     cos(A, B) = (A · B) / (||A|| * ||B||)
     ```

2. **点积（Dot Product）**：
   - 向量对应元素相乘再相加
   - 如果向量已归一化，等价于余弦相似度

3. **欧氏距离（Euclidean Distance）**：
   - 向量空间中两点的直线距离
   - 越小越相似

**RAG 中通常用余弦相似度。**

---

### Q30: 什么是 BM25？和向量检索有什么区别？

**参考答案：**

**BM25**：基于词频的检索算法，是 TF-IDF 的改进版。

**核心思想：**
1. 计算查询中每个词在文档中的权重
2. 综合所有词的权重得到文档得分
3. 返回得分最高的文档

**BM25 的特点：**
- 精确匹配关键词
- 不理解语义
- 适合关键词明确的查询

**BM25 vs 向量检索：**

| 维度 | BM25 | 向量检索 |
|------|------|---------|
| **原理** | 词频统计 | 语义相似度 |
| **同义词** | ❌ 不理解 | ✅ 理解 |
| **精确匹配** | ✅ 好 | ⚠️ 可能错过 |
| **依赖数据** | 不依赖 | 需要 Embedding 模型 |
| **计算速度** | 快 | 相对较慢 |

**最佳实践：混合检索（Hybrid Search）**
- 同时用 BM25 和向量检索
- 然后融合结果

### Q31: 简单介绍一下 Transformer？

**参考答案：**

**Transformer** 是 2017 年 Google 提出的架构，是现在所有大模型的基础。

**核心组件：**

1. **Encoder（编码器）**：
   - 处理输入序列
   - Self-Attention（自注意力）
   - Feed-Forward Neural Network

2. **Decoder（解码器）**：
   - 生成输出序列
   - Masked Self-Attention（掩码自注意力）
   - Cross-Attention（交叉注意力）

**关键概念：**

**Self-Attention（自注意力）**：
- 让每个位置关注到序列的其他位置
- 计算方式：Q（查询）、K（键）、V（值）
- 公式：`Attention(Q, K, V) = softmax(QK^T / √d_k) V`

**为什么用 Transformer？**
- 并行计算（相比 RNN）
- 长距离依赖建模能力强
