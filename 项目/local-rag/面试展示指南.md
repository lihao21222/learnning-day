## 💡 技术亮点详解

### 亮点1：模块化设计
**为什么这么做？**
- 单一职责原则，每个模块只做一件事
- 便于单元测试
- 便于后续扩展和维护

**代码体现**：
```python
# advanced_rag.py 中的类设计
DocumentLoader   # 负责文档加载
TextSplitter     # 负责文本分割
SimpleRetriever  # 负责检索
RAGSystem        # 统一调度
```

**面试话术**：
"我在设计时采用了模块化的思想，把文档加载、文本分割、检索等功能都拆分成独立的类。这样做的好处是，以后如果我想换一个更好的检索算法，只需要修改 SimpleRetriever 这一个类就可以了，不会影响其他部分的代码。"

---

### 亮点2：配置管理
**为什么这么做？**
- 硬编码不利于维护
- 不同环境（开发/生产）需要不同配置
- 方便调整参数

**代码体现**：
```python
# config.py
class Config:
    CHUNK_SIZE = 500
    CHUNK_OVERLAP = 50
    TOP_K = 3
```

**面试话术**：
"我把所有可配置的参数都放到了 config.py 中，比如文本块的大小、重叠大小、检索返回的数量等。这样如果面试官问我，'如果文档很长怎么办？'，我就可以很自然地说，'我会调整 CHUNK_SIZE 这个参数'，而不需要去改代码逻辑。"

---

- 重叠保证上下文不丢失

**代码体现**：
```python

```

**面试话术**：

## ❓ 常见问题准备

### Q1: 这个项目和直接用 LLM 有什么区别？
**回答思路**：
- 数据安全：本地数据不用上传
- 可更新：添加新文档不用重新训练
- 可溯源：知道答案来自哪里
- 减少幻觉：答案基于实际文档

**示例回答**：
"最大的区别在于，直接用 LLM 的话，它的知识是固定的，而且企业内部数据它也不知道。用 RAG 的话，我们可以把自己的文档加进去，而且可以随时更新，不用重新训练。另外，RAG 还可以告诉我们答案来自哪个文档，这样我们可以去验证，也更可信。"

---

### Q2: 如果检索不到相关内容怎么办？
**回答思路**：
- 承认这是个常见问题
- 给出多个解决方案
- 展示你的思考深度

**示例回答**：
"这是个很好的问题！实际中确实会遇到。我觉得可以从这几个方面优化：

1. **优化文本分割**：可能是 chunk size 设置得不合适，可以调整一下
2. **混合检索**：把关键词检索和语义检索结合起来
3. **重排序**：先用宽松的条件召回更多结果，再用一个小模型重新排序
4. **提示用户**：告诉用户没有找到，可以尝试换个问法

我目前这个项目是简化版，后续我打算加一个重排序的步骤，用 BGE-Reranker 这样的模型，应该能提升效果。"

---

### Q3: 这个项目有什么可以改进的地方？
**回答思路**：
- 不要说项目不好
- 展示你有深入思考
- 给出具体的改进方向

**示例回答**：
"这个项目还有很多可以改进的地方：

1. **增加 Web UI**：用 Streamlit 或 Gradio 做一个网页界面，这样演示起来更直观
2. **支持更多格式**：现在只支持 TXT，后续可以加 PDF、DOCX，甚至 PPT、网页
3. **多轮对话**：现在是单轮的，可以加对话历史管理
4. **更好的 LLM**：现在是简化版，可以集成 Ollama 或者 API 来获得更好的回答
5. **评估体系**：加一些自动化测试，来评估检索和回答的质量

其实我都已经想好了怎么实现，比如 Web UI 的话，我会用 Streamlit，因为它做数据应用特别快..."

---

### Q4: 你在做这个项目时遇到的最大困难是什么？
**回答思路**：
- 选一个真实但不难的问题
- 重点说你怎么解决的
- 展示你的学习能力

**示例回答**：
"我遇到的最大困难是一开始不知道怎么把各个模块拼起来。最开始我想直接用 LangChain 的完整链条，但后来发现太复杂了，不利于理解。

于是我换了个思路，先从最简单的版本开始，先做一个纯关键词匹配的版本跑通，然后再逐步加上向量检索、文本分割这些功能。

这个过程让我意识到，做项目不一定要一开始就用最复杂的方案，先用简单的方式跑通，然后再迭代优化，这样反而更快。"
