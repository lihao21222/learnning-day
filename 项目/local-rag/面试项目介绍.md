# 📚 本地知识库问答系统（RAG）

> 一个适合面试展示的 AI 应用项目

---

## 🎯 项目概述

### 项目背景
随着大语言模型的发展，如何让 AI 基于企业内部知识进行准确回答成为关键问题。本项目实现了一个**基于检索增强生成（RAG）的本地知识库问答系统**，可以加载本地文档，建立语义索引，并提供智能问答功能。

### 核心功能
- 📄 **多格式文档支持** - 支持 .txt, .pdf, .docx 文档
- 🔍 **智能语义检索** - 基于向量相似度的语义搜索
- 💬 **自然语言问答** - 基于检索结果生成自然语言回答
- 💾 **持久化存储** - 向量数据库保存到本地

## 🛠️ 技术架构

### 技术栈
| 层级 | 技术选型 | 说明 |
|------|---------|------|
| 编程语言 | Python 3.x | 主流 AI 开发语言 |
| 文档加载 | LangChain | 统一文档加载接口 |
| 文本分割 | RecursiveCharacterTextSplitter | 智能文本切分 |
| 向量模型 | sentence-transformers | 中文语义嵌入 |
| 向量数据库 | Chroma | 轻量级本地向量库 |
| LLM | Hugging Face 模型 | 开源大语言模型 |

### 系统架构
```
用户问题
    ↓
[ 查询处理 ]
    ↓
[ 向量检索 ] ←───── [ 向量数据库 ]
    ↓                    ↑
[ 上下文构建 ]         [ 文档处理 ]
    ↓                    ↑
[ LLM 生成回答 ]    [ 文档加载 ]
    ↓
返回答案 + 引用来源
```

## 🎓 项目亮点（面试重点）

### 1. RAG 核心原理
**解决的问题**：
- LLM 知识有截止日期，无法获取最新信息
- 企业内部知识不在 LLM 训练数据中
- 直接 fine-tune 成本高，且难以更新

**RAG 的优势**：
- 无需训练，实时更新知识库
- 可以溯源，回答有据可依
- 降低幻觉（Hallucination）问题

### 2. 文本分割策略
**为什么重要**：
- 向量模型有输入长度限制
- 太长的文本会丢失语义信息
- 太短的文本上下文不完整

**实现方案**：
- 按句子边界分割
- 设置合理的 chunk size（如 500 字符）
- 添加 overlap（如 50 字符）保持上下文连贯性

### 3. 检索优化
**实现的策略**：
- 关键词匹配 + 语义相似度混合
- Top-K 检索
- 结果按相关性排序

### 4. 模块化设计
- 文档加载器、向量存储、问答模块分离
- 易于维护和扩展
- 符合软件工程最佳实践

## 💡 面试常见问题准备

### Q1: 为什么选择 RAG 而不是 fine-tune？
**回答要点**：
- 数据安全：内部数据不用上传
- 成本更低：不需要 GPU 训练
- 更新灵活：添加新文档不用重新训练
- 可溯源：可以知道答案来自哪里

### Q2: 如何解决检索不到相关内容的问题？
**回答要点**：
- 优化文本分割策略
- 调整 chunk size 和 overlap
- 使用混合检索（关键词 + 语义）
- 增加重排序（Rerank）步骤

### Q3: 如何评估 RAG 系统的效果？
**回答要点**：
- 检索准确率：相关文档是否在 Top-K 中
- 回答准确率：答案是否正确
- 忠实度：答案是否基于检索内容
- 用户满意度：实际使用体验

### Q4: 项目有哪些可以优化的方向？
**回答要点**：
- 加入重排序模型（如 BGE-Reranker）
- 支持更多文档格式（PPT, Excel, 网页等）
- 增加对话历史管理
- 加入 Web UI（如 Streamlit, Gradio）
- 优化性能（批量处理、缓存等）

## 🎨 简历中的项目描述

**项目名称**：基于 RAG 的本地知识库问答系统
